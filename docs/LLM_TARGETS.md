# LLM Targets

## 1. Purpose

AgentFlow should support several output targets because different coding agents use different project context files.

The source stays the same. The generated target changes.

```txt
.agent/src/project.afx -> target emitters -> AGENTS.md, CLAUDE.md, GEMINI.md
```

## 2. Codex target

Output:

```txt
AGENTS.md
```

Use this for OpenAI Codex and tools that recognize `AGENTS.md` style project instructions.

Recommended generated sections:

```md
# Agent Instructions
## Project
## Stack
## Rules
## Commands
## Workflows
## Permissions
## Response Format
```

Codex oriented wording should be direct and operational.

## 3. Claude target

Output:

```txt
CLAUDE.md
```

Use this for Claude Code project memory.

Recommended generated sections:

```md
# Claude Code Project Memory
## Project Context
## Persistent Instructions
## Commands
## Workflow
## Response Format
```

Claude oriented wording can be slightly more natural language and memory focused.

Optional import mode: instead of duplicating content, `CLAUDE.md` can be emitted as a thin file that `@import`s `AGENTS.md` (Claude Code supports `@path` imports) plus Claude-specific notes. The default may remain full emit; both modes are supported.

## 4. Gemini target

Output:

```txt
GEMINI.md
```

Use this for Gemini CLI project context.

Recommended generated sections:

```md
# Gemini Project Context
## Project
## Coding Style
## Commands
## Task Workflow
## Safety Rules
```

## 5. Cursor target (Later / MVP-plus)

This target is not part of MVP. MVP ships Codex, Claude, Gemini, and Runtime JSON only.

Output:

```txt
.cursor/rules/agentflow.mdc
```

Recommended generated format:

```mdc
---
description: AgentFlow generated project instructions
alwaysApply: true
---

Project instructions here.
```

## 6. Runtime target

Outputs:

```txt
.agent/dist/manifest.json
.agent/dist/policy.json
.agent/dist/workflows/*.json
.agent/dist/schemas/*.json
```

This target is for active runtime enforcement and future integrations.

## 7. Skills target, later

Possible output:

```txt
.agent/skills/coder/SKILL.md
.agent/skills/reviewer/SKILL.md
```

Use for packaged specialized capabilities.

## 8. Target selection

CLI examples:

```bash
agentflow build --target codex
agentflow build --target claude
agentflow build --target gemini
agentflow build --target runtime
agentflow build --target all
```

Config example:

```ts
export default defineConfig({
  targets: ["codex", "claude", "gemini", "runtime"]
});
```

## 9. Target adapter interface

```ts
interface TargetAdapter {
  id: string;
  displayName: string;
  outputPaths: string[];
  emit(input: TargetInput): Promise<GeneratedFile[]>;
}
```

## 10. Target specific caveats

### Generated markdown is not enforcement

Even if generated markdown says not to access `.env`, the host tool must enforce that. If not, it remains guidance.

### Different tools load context differently

Each tool may have different rules for file discovery, precedence, and nested instructions. AgentFlow should document target behavior and avoid relying on complex precedence for MVP.

### Keep generated files readable

The generated file should be understandable when opened directly.

Bad:

```md
RULE_001: STRICT_TS
RULE_002: RBW
```

Good:

```md
- Use strict TypeScript.
- Read a file before editing it.
```

## 11. Target metadata

Generated files should include source notice:

```md
Generated by AgentFlow from .agent/src/project.afx. Do not edit directly.
```
